# Организация A/B-тестирования

1. ## Концепция A/B-теста

Делим трафик случайным образом, 50% направляем в  модель v1, 50% в модель v2 случайным образом. Это может нам обеспечить сопоставимость групп по составу клиентов.
Продолжительность экспериментов 1-2 месяца, но зависит от скорости наполнения данных .
2. ## Метрики и анализ

Основная метрика -  определение положительного класса дефолта F1-score для класса дефолта, так как класс дефолта встречается более редко. 
Дополнительная метрика — RECALL для класса дефолта, что бы понять какую долю «аварийных» клиентов мы отловили.

3. ## Метод статистического анализа

Мы знаем, что группы независимые: пользователи в группах А и B не пересекаются. Значит, мы можем воспользоваться односторонним Z- тестом для пропорций для Recall.
Так метрика F1 составная, то для ее анализа лучше применять бутстрап для оценки распределения разницы F1-score(v2) — F1-score(v1);
Доверительные интервалы
Для оценки устойчивости результата строятся 95% доверительные интервалы.
Для Recall
Можно можно построить классический доверительный интервал для разности пропорций.
Для F1-score
Предлагается использовать бутстрап-доверительный интервал.
Новая модель v2 считается успешной, если выполняются одновременно следующие условия:
Основная метрика улучшилась:
F1-score(v2) > F1-score(v1)
Recall(v2) >= Recall(v1)
новая модель не должна приводить к росту ожидаемых финансовых потерь.
