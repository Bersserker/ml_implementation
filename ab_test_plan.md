## Организация A/B-тестирования

### Цель теста

Цель A/B-теста — сравнить две версии модели кредитного скоринга:

* **контрольная группа (A)** — текущая модель `v1`;
* **тестовая группа (B)** — новая модель `v2`.

Задача теста — понять, улучшает ли новая модель качество выявления дефолтных клиентов без недопустимого ухудшения других важных показателей.

---

## Схема проведения теста

### Разделение трафика

Трафик предлагается делить **случайным образом 50/50**:

* 50% заявок направляются в модель `v1`;
* 50% заявок направляются в модель `v2`.

Случайное распределение нужно для того, чтобы группы были сопоставимы по составу клиентов и внешним условиям.

Для практической реализации можно использовать:

* случайный выбор по `request_id` или `client_id`;
* хеширование идентификатора клиента с фиксированным правилом распределения.

Это позволит:

* избежать ручного перекоса;
* обеспечить повторяемость распределения;
* не допускать, чтобы один и тот же клиент в пределах теста попадал то в одну, то в другую группу.

---

## Продолжительность теста

Продолжительность A/B-теста зависит от объёма заявок и скорости накопления размеченных исходов, но в концепции проекта можно принять:

* **минимальная длительность — 2–4 недели**,
  или
* до достижения заранее заданного объёма наблюдений в каждой группе.

Так как в кредитном скоринге факт дефолта определяется не мгновенно, в реальной системе тест может длиться существенно дольше.
Для учебного описания разумно указать, что тест завершается, когда:

1. собран достаточный объём заявок в обеих группах;
2. по ним доступны фактические целевые исходы;
3. достигнута статистическая мощность, достаточная для сравнения моделей.

---

## Метрики и анализ

### Основная метрика: F1-score для класса дефолта

Основной метрикой выбирается **F1-score для положительного класса**, то есть для класса дефолта.

#### Почему именно F1-score

В задаче кредитного скоринга класс дефолта обычно является более редким, чем класс недефолта.
При этом важны сразу два аспекта:

* не пропустить дефолтных клиентов;
* не давать слишком много ложных тревог.

F1-score является гармоническим средним между Precision и Recall и позволяет учитывать оба типа ошибок одновременно.

Формула:

[
F1 = 2 \cdot \frac{Precision \cdot Recall}{Precision + Recall}
]

---

### Дополнительная метрика: Recall для класса дефолта

В качестве дополнительной метрики предлагается использовать **Recall** для класса дефолта.

#### Почему выбран Recall

Recall показывает, какую долю реально дефолтных клиентов модель смогла обнаружить:

[
Recall = \frac{TP}{TP + FN}
]

Для кредитного скоринга это особенно важно, потому что:

* **ложноотрицательная ошибка** означает, что дефолтный клиент был признан надёжным;
* такая ошибка напрямую связана с финансовыми потерями.

Поэтому даже если F1-score улучшается, важно отдельно контролировать Recall, чтобы новая модель не пропускала больше рискованных клиентов.

---

### Возможная альтернативная дополнительная метрика

В некоторых бизнес-сценариях вместо Recall можно использовать **Precision**, если для заказчика особенно важно уменьшить число клиентов, ошибочно помеченных как дефолтные.
Но для кредитного риска чаще более критичен именно Recall, так как пропущенный дефолт обычно дороже ложного отказа хорошему клиенту.

---

## Статистический анализ

### Что сравнивается

После завершения теста для групп A и B вычисляются:

* F1-score для дефолта;
* Recall для дефолта.

Сравнение проводится на одинаковом временном окне и на сопоставимых по распределению данных.

---

### Статистический тест

Для сравнения долей и метрик, рассчитанных на больших выборках, можно использовать:

* **z-test для сравнения пропорций** — для метрик, основанных на долях, например Recall;
* для более общей оценки различий между метриками по бутстрап-выборкам — **t-test** по распределению метрики на батчах или бутстрап-итерациях.

В практическом описании для данного проекта удобно зафиксировать следующий подход:

#### Для Recall

Используется **z-test для двух пропорций**, так как Recall можно интерпретировать как долю найденных дефолтов среди всех реальных дефолтов.

#### Для F1-score

Так как F1-score — составная метрика и её аналитическое распределение менее удобно, предлагается использовать:

* **бутстрап** для оценки распределения разницы `F1(v2) - F1(v1)`;
* далее строить доверительный интервал для этой разницы.

Это более надёжный и понятный способ для прикладного ML.

---

## Доверительные интервалы

Для оценки устойчивости результата строятся **95% доверительные интервалы**.

### Для Recall

Можно строить:

* либо классический доверительный интервал для разности пропорций;
* либо бутстрап-интервал.

### Для F1-score

Предлагается использовать **бутстрап-доверительный интервал**:

1. многократно случайно переотбирать объекты с возвращением внутри каждой группы;
2. на каждой итерации пересчитывать `F1_A` и `F1_B`;
3. строить распределение разности `F1_B - F1_A`;
4. брать 2.5 и 97.5 процентили как границы 95% доверительного интервала.

---

## Критерий успешности теста

Новая модель `v2` считается успешной, если выполняются одновременно следующие условия:

1. **Основная метрика улучшилась**:
   `F1-score(v2) > F1-score(v1)`

2. **95% доверительный интервал для разницы F1-score не включает 0**
   То есть улучшение статистически подтверждено.

3. **Recall не ухудшился ниже допустимого порога**
   Например:

   * либо `Recall(v2) >= Recall(v1)`;
   * либо снижение Recall не превышает заранее согласованный порог, например 1–2%.

4. При наличии бизнес-оценки риска:

   * новая модель не должна приводить к росту ожидаемых финансовых потерь.

---

## Пример формулировки критерия принятия

Тест считается успешным, если модель `v2` показывает статистически значимое улучшение F1-score для класса дефолта по сравнению с `v1`, а Recall для дефолта не ухудшается либо остаётся в пределах допустимого бизнес-отклонения.

## Итоговая схема A/B-теста

* **Группа A**: модель `v1`
* **Группа B**: модель `v2`
* **Разделение трафика**: случайное, 50/50
* **Длительность**: 2–4 недели или до накопления достаточной выборки
* **Основная метрика**: F1-score для класса дефолта
* **Дополнительная метрика**: Recall для класса дефолта
* **Статистический анализ**:

  * z-test для Recall;
  * бутстрап и доверительный интервал для разницы F1-score
* **Критерий успеха**:

  * F1-score у `v2` выше, чем у `v1`;
  * доверительный интервал улучшения не включает 0;
  * Recall не ухудшается сверх допустимого порога

## A/B-тестирование моделей кредитного скоринга

### Цель

Сравнить текущую модель (**v1**, контрольная группа) и новую модель (**v2**, тестовая группа) с точки зрения качества выявления дефолтных клиентов и влияния на бизнес-показатели.

---

### Дизайн эксперимента

* **Разделение трафика:** случайное, 50/50
  (по `client_id` или `request_id` для воспроизводимости)

* **Группа A:** модель v1

* **Группа B:** модель v2

* **Продолжительность:**
  2–4 недели или до достижения статистически значимого объёма данных

---

### Метрики

#### Основная метрика

* **F1-score для класса дефолта**

Обоснование: учитывает баланс между Precision и Recall, что важно при несбалансированных классах.

---

#### Дополнительная метрика

* **Recall для класса дефолта**

Обоснование: минимизация пропущенных дефолтов, которые напрямую ведут к финансовым потерям.

---

### Статистический анализ

* **Recall:**
  используется **z-test для сравнения пропорций**

* **F1-score:**
  используется **бутстрап-оценка разности метрик**:

  * генерируются выборки с возвращением;
  * вычисляется распределение `F1(v2) - F1(v1)`

---

### Доверительные интервалы

* Строится **95% доверительный интервал** для разницы метрик:

  * для Recall — аналитически или через бутстрап;
  * для F1 — через бутстрап (перцентили 2.5% и 97.5%).

---

### Критерий успешности

Модель **v2** принимается, если выполняются условия:

1. **F1(v2) > F1(v1)**
2. 95% доверительный интервал для `F1(v2) - F1(v1)` **не включает 0**
3. **Recall(v2)** не ухудшается или снижается не более чем на допустимый порог
4. (опционально) не увеличиваются ожидаемые финансовые потери

---

### Вывод

A/B-тест позволяет объективно сравнить модели в реальных условиях и принять решение о внедрении новой версии на основе статистически значимых улучшений ключевых метрик.
